# Gianluigi v2.5 — Strategy & Architecture

**Status:** Agreed direction (brainstorm 2026-05-21). Not yet implemented.
**Supersedes/absorbs:** the committed v2.4 "agentic synthesis" roadmap — its query/synthesis layer is folded in here as the *read-back loop* + *weekly synthesis* of the knowledge architecture.
**Born from:** ~2 months of live production use. This document captures the diagnosis, the decisions locked with Eyal, the reframed architecture, and the phased roadmap.

> This is forward-looking strategy. For what the code does *today*, trust `git log`, `KNOWN_ISSUES.md`, and the source. For the v1 spec, see `V1_DESIGN.md`.

---

## 1. Diagnosis (why we're changing course)

After two months of real use:

1. **Meeting summary & distribution** — the strongest part, relied on daily. But it misses points and underuses prior knowledge.
2. **Knowledge development is shallow.** The system does near-isolated analysis of each meeting with a thin lookup, rather than developing an evolving understanding.
3. **The outputs (tasks / Gantt / follow-ups / prep)** are mostly features that were *buildable*, not features that map to daily decisions — so they go unused.
4. **Intelligence report** delivery is unreliable and its quality is generic.
5. **Google Workspace** migration is now possible (company got a domain) and could upgrade capture.

**Root technical finding:** the three things that should form a learning loop — semantic memory (embeddings/RAG), evolving topic-state (`topic_threads.state_json`), and meeting extraction — barely talk to each other:
- Embeddings/RAG are built but **not used during extraction** (only for manual queries).
- The extraction prompt literally instructs the model to **drop detail** ("aim for 3-7 action items, not 10-15").
- Extraction is **single-pass**, with shallow prior context (2-3 recent participant-overlap meetings, ~3,000 chars).
- Topic-state evolves only weakly (fire-and-forget Haiku merge, post-approval), is **never re-synthesized**, and is **not read back into extraction**.

The outputs' friction is structural: **Google Sheets is used as both the database and the UI**, which produces the sync fragility, silent data loss, and the brittle Gantt-status-from-cell-colors.

---

## 2. The reframe

From **"an assistant with many features"** → **"a company brain you can interrogate."**

- **Knowledge layer is the spine.** It is continuously developed, consolidated, and — critically — **read back** into extraction and the push-brief.
- **Push-primary, pull-secondary.** The morning brief becomes the brain's voice; "ask anything" is the secondary CEO-walks-up path.
- **Communication like a human assistant.** Reaches Eyal on the right channel for the moment, text and voice equal, switches on command, falls back gracefully.
- **Database is the source of truth; UI is a swappable, downstream view.** Editing any output (manually or via chat/voice) flows back to the DB.

---

## 3. Locked decisions

- **Phase order:** Knowledge foundation **first**; everything else reads from it.
- **Cost posture:** willing to run ~2-3x current LLM/infra spend where value is clearly proven; optimize later. Use cheap models (Haiku) for routine passes, premium (Sonnet/Opus) for high-stakes synthesis.
- **Rollout:** **shadow-run** the new knowledge layer alongside the live system, compare, cut over when proven. Daily flow never breaks.
- **Build forward, never from scratch.** Bootstrap all new structure from existing data (meetings, embeddings, topic_threads, canonical_projects, labels, Gantt). Migrations are **additive + backfill-friendly**; never wipe.
- **Graph approach:** **graph-lite on existing Postgres** — adopt bi-temporal facts + explicit typed links; do **not** add a separate graph DB (Graphiti/Zep) yet. Revisit when *data volume* (not headcount) demands.
- **Gantt:** shares the knowledge taxonomy/naming, but the dependency is **one-way** (knowledge → renders to → Gantt). The knowledge workflow never reads from or waits on the Gantt sheet.
- **Headcount-agnostic:** roster is data-driven; no hardcoded team members, owner-prefixes, or "N members = CropSight" heuristics. Tiers/distribution scale to any team size.
- **Topic curation:** semantic clustering **proposes** new topics/merges for approval; Eyal can also **edit topics himself** on demand (and those edits double as preference-learning signal).
- **Autonomy:** quiet internal-understanding updates (on-event/nightly brief updates) run automatically; **structural moves** (merging topics, superseding decisions) and anything outbound are gated on Eyal's approval.
- **Nothing gets cut.** Gantt, prep, follow-up tracking all stay — re-architected for quality, not removed.

## 4. The knowledge architecture (centerpiece)

### 4.1 Three memory types (the organizing vocabulary)

| Memory type | What it is | Maps to |
|---|---|---|
| **Episodic** | What happened | Raw events: meetings, emails, docs, injections (have it) |
| **Semantic** | What we *know* | Atomic items → topic briefs → area briefs → company picture (thin today) |
| **Procedural** | How *I* work & how Eyal likes it | **Missing today** — this is the "learning" layer |

### 4.2 The five-layer hierarchy (semantic memory)

1. **Raw events** — immutable source record (episodic).
2. **Atomic items** — tasks, decisions, open questions, stakeholders (with lifecycle).
3. **Topic briefs** — one living page per initiative (the `state_json`, made rich and actually read).
4. **(3.5) Area / sphere briefs** — aggregate topic briefs. Aligned with Gantt naming. Proposed CropSight spheres (align to real Gantt):
   - Product & Technology / R&D
   - Client Delivery & Operations
   - Business Development & Commercial
   - Corporate, Legal & Finance
   - Company & Team
5. **Company picture** — synthesis across spheres; the strategic state.

Hierarchy: **Area → Topic → Sub-topic → Items.** Designed properly from the ground up (taxonomy), reusing existing data.

### 4.3 Three maintenance cadences

| Cadence | Trigger | Does | Model |
|---|---|---|---|
| **On-event** | A meeting/email/injection mentions a topic | Merge new info into the brief; flag changes; update items | Haiku |
| **Nightly** | Scheduled sweep over touched topics | Reconcile contradictions, recompute status, age items, catch what on-event missed | Haiku/Sonnet |
| **Weekly** | Scheduled, all active topics | Re-synthesize each brief from full history; cross-topic patterns; strategic narrative | Sonnet/Opus |

(This is Complementary Learning Systems: fast episodic capture → periodic consolidation into semantic understanding.)

### 4.4 The three "verbs" + the additional capabilities

- **Develop** — understanding grows (events accrete into briefs; new topics spawn; links form).
- **Consolidate** — stays clean (merge duplicates, resolve contradictions, age out stale, re-synthesize).
- **Update** — existing understanding changes (state transitions, supersessions).

On top of those maintenance verbs:
- **Recall / Retrieval** — read knowledge *back* into extraction and queries (the loop, §5).
- **Reflection** — periodic self-critique ("what's inconsistent / stale / untouched / where was I corrected?") feeding proactive flags.
- **Learning (procedural — the new layer):**
  - *Preference learning* from Eyal's edits/approvals/rejections (learn tone, what to gate, channel prefs).
  - *Quality learning* from corrections (store error patterns / few-shot to improve future extraction).
- **Foresight & Connection** — anticipate slippage/idle topics/decisions-due; infer cross-topic implications.
- **Forgetting / decay** — down-weight stale info (recency/relevance/salience) so old noise doesn't drown signal.

### 4.5 Storage: graph-lite on Postgres

- **Bi-temporal facts** — every belief carries "true since / superseded on"; old facts are **invalidated, not deleted**. Generalizes existing decision-supersession + topic-state versioning.
- **Typed links** — a lightweight relations table: `topic belongs_to area`, `decision supersedes decision`, `task advances topic`, `topic blocks topic`. This is the "graph" (connection/inference) without a graph DB.
- **Sensitivity & provenance carry through** every new layer (tier + source citations).

## 5. The read-back loop (fixes points 1 & 2)

Today knowledge flows one way (events → storage, then it sits). The fix — make it flow back to two consumers:

1. **Into extraction (the big one).** Before analyzing a new meeting, retrieve relevant topic/area briefs + semantically-similar past content (RAG) and feed them into the prompt. Extraction becomes *informed*: knows a task keeps slipping, a decision contradicts March, Paolo already owns something similar. **Also remove the "aim for 3-7 items" muzzle and add a one-shot completeness check** against open commitments.
2. **Into the push-brief.** Morning brief reads topic/area states → surfaces "Legal blocked 12 days," "decision X overdue for review," proactively.

## 6. Interaction & communication (human-assistant)

**North star:** behave like a great human EA. Text (read *and* write) stays first-class; voice is an added lane; switches channel on command ("ping me on WhatsApp"); falls back gracefully.

**Three parts:**
- **Channels:** Telegram (have; supports voice notes), WhatsApp (new — note 24h-window/template constraint for proactive push), voice call (new), email (have), Claude/Live-Artifact cockpit (pull).
- **Modalities:** text in/out (have); voice-in (STT, ~$0.006/min); voice-out (TTS — ElevenLabs already integrated); real-time two-way voice (hardest).
- **Orchestration spine** ("reach Eyal" router): picks channel by context/time/urgency/preference, falls back, switches on command, remembers preferences. The thing that makes it feel human.

**Injection gap = keystone.** Eyal won't type injections but will talk.
- *Eyal-initiated:* record a voice note (or type) any time → injected.
- *System-initiated:* the evening debrief becomes a reach-out ("how did today go?"); answer by voice or text; it asks follow-ups, extracts, injects.

**Voice build ladder:** (1) voice notes on Telegram → (2) orchestration spine → (3) WhatsApp channel → (4) voice-out → (5) conversational evening debrief *call*. First beat = **#1 + #2 together**.

## 7. Outputs & UI

- **DB = source of truth; bidirectional edit** (manual edits flow back to DB).
- **UI = hybrid:** keep **Google Sheets as the team-facing, mobile, no-login view** now (DB authoritative, sheet a reliable generated mirror — kills the sync-fragility class). Layer a **Claude Live Artifact** as Eyal's private interactive cockpit later (read/write DB via MCP), piloted on one surface once the MCP-in-artifact bug clears. Push always lives on conversational channels (artifacts can't push).
- **Gantt** restructured to the Area→Topic taxonomy; becomes a *time-view render* of the knowledge layer (one-way).
- **Follow-up tracking** and **meeting prep** re-architected as *views/actions off the knowledge layer* (to be characterized in their own phases) — surfaced proactively, with quality + interaction as the bar.

**Live Artifacts research caveats:** can't push/run in background; team sharing is walled (non-Claude users blocked, public publish disabled on Team/Enterprise); MCP-in-artifact has a live param-stripping bug + per-user re-auth; reliability not yet production-grade. Hence "cockpit later," not "backbone now."

## 8. Capture expansion

- **Voice + manual injection** (see §6) — closes the non-recorded-meeting gap.
- **Email & documents** — the watchers already exist but only *file* things. Re-point existing ingestion to feed the **knowledge layer** (topic briefs) and surface in the brief, instead of archiving silently.
- **Workspace migration (sequenced later)** — 5 services move from per-user OAuth refresh tokens → service account + domain-wide delegation (code already has "Phase B" scaffolding; tests use `@cropsight.com`). Capability unlock: **Meet + Gemini "take notes for me"** gives per-speaker attribution natively (each joins on own laptop, Adaptive Audio) — could replace Tactiq and fix the multi-speaker-in-room + split-recording problems.

## 9. Sequenced later (not dropped)

- **Workspace migration** — foundation + better capture; do after the knowledge foundation lands.
- **Intelligence signal** — fix delivery (verify `INTELLIGENCE_SIGNAL_ENABLED`, the restart-unsafe 30-min `asyncio.sleep` in distribution), then re-aim quality at CropSight's real open questions/decisions. Open question to resolve in this phase: **does anyone watch the video?** (expensive/fragile; measure before keeping).

## 10. Cross-cutting principles (non-negotiable)

- DB is source of truth; UI is downstream + swappable; edits are bidirectional.
- Headcount-agnostic (data-driven roster; no hardcoded team).
- Gantt dependency is one-way (knowledge → Gantt).
- Sensitivity follows data through every new layer; source citations preserved.
- Restart-safety: anything mid-flow must be replayable from DB rows (audit/replace in-memory timers like the 30-min sleep).
- Additive migration; build forward from existing data; never wipe.
- Gianluigi proposes, Eyal approves (structural/outbound gated; quiet internal updates auto).

## 11. Phased roadmap

### Phase 1 — Knowledge foundation (FIRST)
Build the spine everything reads from, shadow-run alongside live.
1. Schema (additive): area/topic/sub-topic hierarchy; enriched living briefs; bi-temporal fields; typed-links table. Backfill from existing meetings/embeddings/topic_threads/labels.
2. One-time historical synthesis pass → initial topic + area briefs (cold-start solved from 2 months of data).
3. Semantic clustering → **propose** Areas/topics for approval; align naming to the Gantt.
4. The **read-back loop**: RAG-fed extraction + remove the "3-7" muzzle + completeness check.
5. Cadences: reliable on-event update; new nightly consolidation; new weekly synthesis (with reflection).
6. Shadow-run: compare new briefs/extraction against the live path; cut over when proven.

### Phase 2 — Push-brain + comms keystone
- Morning brief reads area/topic state (foresight flags).
- Voice-note injection (#1) + orchestration spine (#2).

### Phase 3 — Outputs re-architecture
- DB-truth + bidirectional sync; Sheets as generated team view; Gantt restructured to taxonomy.
- Characterize + rebuild follow-up tracking and meeting prep as knowledge-layer views.

### Phase 4 — Comms depth + cockpit
- WhatsApp channel; voice-out; conversational evening debrief call.
- Live Artifact pilot (tasks) as Eyal's cockpit.

### Phase 5 — Workspace + intelligence signal
- Migrate to Workspace (service account + domain-wide delegation); evaluate Meet+Gemini capture.
- Fix intelligence-signal delivery + restart-safety; re-aim quality; decide video fate.

### Cross-cutting hygiene (throughout)
- Restart-safety audit; tool-sprawl review (44 MCP tools); get the ~21 failing baseline tests green; graduated-autonomy candidates from approval-observation data.

## 12. Success signals (so we don't build more shelfware)

- **Extraction completeness** — meeting summaries capture the action items Eyal would have noted (periodic spot-audit + his corrections trend down).
- **Injection usage** — # voice/text injections per week (is he actually using it?).
- **Push relevance** — morning brief surfaces things he acts on (thumbs up/down).
- **Knowledge accuracy** — "where are we on X?" returns a correct synthesized answer (correction rate).
- **Output engagement** — he actually uses the Gantt/tasks views.

Eyal's feedback is both the success metric *and* the procedural-learning signal — they're the same stream.
