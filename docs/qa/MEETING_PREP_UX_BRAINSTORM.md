# Meeting Prep — UX & Flow Brainstorming Notes

**Context:** Phase 5 is code-complete (1067 tests). During initial live testing we noticed several friction points in the meeting prep experience. These are thinking points and proposals — not final decisions. Consider these alongside the current implementation and propose what makes sense to change, what to keep, and what to defer.

---

## Core Observation

The current prep flow treats Telegram and Drive as the same output channel. It generates one Markdown document and sends it everywhere — resulting in raw Markdown in Telegram, walls of text on a phone screen, and a format that's too detailed for quick scanning but too compressed for a proper briefing document. The flow should recognize that **Telegram is for deciding, Drive is for reading.**

## Proposals to Consider

### 1. Two distinct output formats

Instead of one Markdown blob sent everywhere, consider splitting into:

- **Telegram briefing card** — 10-15 lines of plain text (no Markdown formatting). Optimized for scanning on a phone. Generated by Haiku (cheap, fast). Includes a link to the full doc in Drive.
- **Drive prep document** — Full structured .docx with sections, tables, Gantt snapshots. Generated by Sonnet. This is what gets distributed to the team on approval.

The outline proposal, the approval message, and the reminder messages should all be short plain-text Telegram messages with links — never inline full documents.

### 2. "Since last meeting" as the lead section

The most valuable context before any recurring meeting is: what happened since the last one of this type? The current prep is stateless — it pulls current tasks and decisions without reference to the previous meeting.

Proposal: add a data query like `get_changes_since_last_meeting(meeting_type)` that returns:
- Tasks completed since last meeting of this type
- Tasks newly overdue since then
- New decisions made
- Commitments fulfilled or broken
- New entities/stakeholders that appeared

This requires the `meeting_type` column on `meetings` (already in Phase 5 migration) to be populated during transcript processing. The "since last meeting" narrative should lead the Telegram briefing card — it's the thing Eyal most needs to see.

### 3. Rethink emergency mode (<6h)

Current behavior: sends outline AND starts parallel background generation, creating a race condition that results in two competing Telegram messages. This is confusing UX.

Proposal: for <6h, skip the outline entirely. Generate a **quick brief** — a shorter, punchier format that goes straight to approval. One message, one action. The quick brief has fewer sections (no Gantt snapshot, no full task inventory) — just "3 things to know before this meeting" + approve/reject. This eliminates the race condition and matches reality: with 3 hours until a meeting, there's no time for a discuss-and-adjust cycle.

Think about whether this should be a third format (alongside briefing card and full doc) or just a condensed version of the briefing card.

### 4. Make generic template actually useful

Most meetings currently score <2 on classification and fall to generic, which pulls loosely-related data. The generic template should be participant-centric rather than topic-centric:
- Tasks assigned to meeting participants (by status)
- Most recent decisions involving those participants
- Open commitments by those people
- Any recent email intelligence involving those contacts

This makes even an unclassified meeting's prep useful. Worth investigating WHY most meetings score <2 — is it title inconsistency, participant email format mismatch, or match patterns being too narrow?

### 5. Outline proposal tone and structure

Current outline reads like a data inventory ("Tasks: 5, Decisions: 3, Open Questions: 2"). It should read like a brief from an executive assistant:

```
Prep: Tech Review tomorrow 14:00 (Eyal, Roye, Paolo, Yoram)

Since last Tech Review (Mar 2):
— Roye completed cloud provider evaluation (was overdue)
— Moldova data pipeline still pending (7 days overdue)
— Open Q: "time to stable predictions" — still unresolved

Current state:
— 3 open tasks for Roye (1 overdue)
— Gantt Product & Tech: W12, on track except pipeline
— No new commitments since last meeting

Suggested focus: Moldova pipeline delay, prediction stability

[Full prep doc] [Add focus] [Looks good, generate] [Skip]
```

Plain text, narrative, scannable on a phone. The suggested focus is derived from what's overdue or urgent — not generic boilerplate from the template config.

### 6. Approval message should be minimal

After full generation, the approval message shouldn't contain the document. Just:

```
Prep doc ready: Tech Review (Mar 18, 14:00)
📄 View in Drive: [link]
Sections: Follow-up, Tasks, Gantt, Commercial context (added), Agenda

[Approve + send to team] [Edit] [Reject]
```

The current behavior of sending the full Markdown inline leads to the raw formatting issues and message bloat.

### 7. Meeting-type-specific content focus

Not all templates should pull the same data shape. Think about what's actually useful per type:

- **Technical Review**: Roye's delivery status, technical decisions pending, Gantt Product & Tech section, technical open questions. Lead with: "did Roye deliver what he committed to?"
- **Business Review**: Pipeline movement, Paolo's task status, stakeholder changes, commercial deadlines. Lead with: "what moved in the pipeline?"
- **Monthly Strategic**: Full scorecard — completion rates, Gantt progress vs plan, commitment fulfillment rate. This is the only type that should be comprehensive. The others should be focused and brief.

## Questions for You to Think About

- Does the two-format split (Telegram card vs Drive doc) add too much complexity, or is it a clean separation that simplifies both?
- For the "since last meeting" query: what if there's no previous meeting of that type in the DB yet (first occurrence)? Graceful fallback needed.
- Should the quick brief for emergency mode be a separate template/format, or just the normal briefing card without the outline interaction step?
- The outline proposal rewrite (narrative style) — does this require Sonnet instead of Haiku to generate well, or can Haiku handle a 10-line briefing card?
- Any of these changes that should wait for after live QA stabilization rather than being addressed now?
