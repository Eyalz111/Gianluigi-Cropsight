### Changes 3+4: Meeting Prep + Weekly Digest -> Eyal Approval

Previously, meeting prep docs and weekly digests were auto-distributed directly to the team. Now they route through Eyal's approval flow first — same pattern as meeting summaries.

**Core change in `guardrails/approval_flow.py`:**
- Added `_pending_approvals: dict[str, dict]` — tracks content type per approval (meeting_summary, meeting_prep, weekly_digest)
- `submit_for_approval()` now branches by `content_type`:
  - **meeting_prep** — formats prep-specific approval message, stores sensitivity info
  - **weekly_digest** — formats digest-specific approval message
  - **default** — existing meeting summary flow
- `process_response()` dispatches to correct distributor based on stored content type
- New `distribute_approved_prep()` — sends via Telegram (sensitivity-aware: Eyal-only or full team)
- New `distribute_approved_digest()` — sends via email to team + Telegram group message

**Scheduler changes:**
- `schedulers/meeting_prep_scheduler.py` — replaced direct `telegram_bot.send_to_eyal()`/`send_to_group()` with `submit_for_approval(content_type="meeting_prep", ...)`
- `schedulers/weekly_digest_scheduler.py` — replaced direct `gmail_service.send_email()`/`telegram_bot.send_to_group()` with `submit_for_approval(content_type="weekly_digest", ...)`. Removed unused `gmail_service` and `telegram_bot` imports.

**Test updates:**
- `tests/test_meeting_prep.py` — 2 tests rewritten to verify `submit_for_approval` is called with correct `content_type` and `sensitivity`
- `tests/test_weekly_digest.py` — 3 tests updated to patch through `guardrails.approval_flow` instead of removed scheduler imports

**New tests:** 18 in `tests/test_approval_routing.py`

**Note:** `_pending_approvals` is in-memory. If Gianluigi restarts before Eyal approves, pending data is lost. Acceptable for now; persistent storage is v0.3.

### Change 6: Multi-Layer Inbound Message Guardrails

**New file:** `guardrails/inbound_filter.py`

5-layer security system applied to ALL inbound messages before they reach the agent:

| Layer | Function | What it does |
|-------|----------|-------------|
| 1 — Sender Verification | `verify_sender_telegram()`, `verify_sender_email()` | Checks sender against team registry. Unknown senders get polite deflection. |
| 2 — Topic Relevance | `check_topic_relevance()` | Regex patterns for off-topic (jokes, poems, recipes, personal advice) vs work indicators (cropsight, meeting, task, investor, etc.). Off-topic gets deflected. |
| 3 — Leak Prevention | `check_response_for_leaks()` | Scans outbound responses for sensitive patterns (equity, salary, NDA, API keys, bank accounts, valuations, term sheets). Context-aware: group chat blocks more than DMs. Sensitive content replaced with "[Sensitive -- contact Eyal]". |
| 4 — Output Sanitization | `sanitize_outbound_message()` | Wraps leak check for all outbound messages (not just summaries). Applied to email replies and Telegram responses. |
| 5 — Audit Logging | `log_inbound_interaction()` | Every interaction logged to Supabase `audit_log` — who, from where, was it relevant, what happened. |

**Main entry point:**
```python
async def check_inbound_message(message, sender_id, channel, ...) -> dict:
    # Returns: {allowed, deflection_message, member_id, audit_logged}
```

**Integration points:**
- `services/telegram_bot.py` — `_handle_message()` calls `check_inbound_message()` BEFORE processing. Outbound calls `sanitize_outbound_message()`.
- `schedulers/email_watcher.py` — `_handle_question()` calls `sanitize_outbound_message()` on outbound replies.
- `core/system_prompt.py` — Added `INFORMATION_SECURITY_RULES` section forbidding financial details, legal docs, and credentials in responses. Directs users to "check with Eyal" for sensitive topics.

**New tests:** 31 in `tests/test_inbound_filter.py`

## Bugs Fixed During Implementation

1. **SupabaseClient property setter** — `client.client = mock` failed with `AttributeError: property 'client' has no setter`. Fix: use `object.__setattr__(client, '_client', mock)` to set private attribute directly.
2. **Mock table routing** — Single MagicMock returned same mock for any `.table()` call, so `audit_log` insert overwrote `tasks` insert's `call_args`. Fix: per-table mocks with `side_effect` routing.
3. **Removed imports in weekly_digest_scheduler** — After removing direct `gmail_service`/`telegram_bot` imports (distribution moved to approval_flow), 3 tests failed patching removed attributes. Fix: patch through `guardrails.approval_flow` instead.
4. **Meeting prep test assertions** — 2 tests asserted `send_to_eyal`/`send_to_group` were called directly, but scheduler now calls `submit_for_approval`. Fix: rewrite assertions to verify approval submission.
