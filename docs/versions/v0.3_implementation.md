# v0.3 Phase 1 — Implementation Notes

## What Was Built

v0.3 Phase 1 adds **cross-meeting operational intelligence** to Gianluigi. The system now connects the dots across meetings automatically:

- When a task from Meeting A is discussed in Meeting B, Gianluigi detects it
- When someone says "we finished the dev setup," Gianluigi infers the task status changed
- When an open question gets answered in a later meeting, Gianluigi marks it resolved
- All inferred changes go through Eyal's approval (bundled with the meeting summary)

This covers **Tier 1** (cross-reference intelligence) and **Tier 3** (RAG improvements) from the v0.3 plan. **Tier 2** (entity registry, proactive alerts, commitment tracking) was implemented on 2026-03-01 — see "Phase 2: Tier 2" section below.

---

## Features Implemented

### 1. Task Deduplication (`deduplicate_tasks`)

**Problem:** The same strategic task gets re-stated across multiple meetings ("prepare investor deck" -> "finalize the deck" -> "finish up the investor materials"). Without dedup, this creates 3 separate tasks in the tracker.

**Solution:** Before storing newly extracted tasks, the system compares them against all existing open tasks using Claude Haiku. Each new task is classified as:
- **NEW** — genuinely new task, inserted normally
- **DUPLICATE** — same task with different wording, creates a `task_mention` record instead
- **UPDATE** — same task but the conversation implies a status change

**Conservative by design:** Tasks are founder-level strategic commitments. The prompt instructs Claude to only classify as DUPLICATE/UPDATE when clearly the same underlying work. Different tasks for the same person in the same category are NOT duplicates.

### 2. Task Status Inference (`infer_task_status_changes`)

**Problem:** Eyal's team discusses task progress in meetings, but the task tracker doesn't update. Someone says "the Moldova timeline is done" and the tracker still shows "pending."

**Solution:** After each transcript is processed, Claude Sonnet analyzes the full transcript against all open tasks. It looks for evidence of status changes with explicit confidence levels:
- **HIGH** — explicit statement ("we finished it," "it's signed")
- **MEDIUM** — strong implication ("I'm working through the timeline")
- **LOW** — weak signal (casual mention without status info)

The prompt instructs Claude to be very conservative: "we're making progress" = still in_progress, NOT done.

### 3. Open Question Resolution (`resolve_open_questions`)

**Problem:** Questions raised in one meeting ("what's our pricing model?") get answered in a later meeting, but the open questions list doesn't update.

**Solution:** Claude Haiku compares all open questions against the transcript. When a question is clearly answered, it's flagged for resolution with the answer text and evidence.

### 4. Cross-Reference Orchestrator (`run_cross_reference`)

The `run_cross_reference()` function orchestrates all three analyses and:
- Creates `task_mention` records for all detected connections
- Returns combined results for inclusion in the approval message
- Logs a summary of findings

### 5. Transcript Pipeline Integration

The cross-reference analysis runs as **Step 7b** in the transcript processing pipeline — after the meeting record is created but before tasks are stored. This means:
- Deduplication happens before `create_tasks_batch()` — only genuinely new tasks get inserted
- The `cross_reference` results are passed through to the approval flow

### 6. Approval Flow Integration

Cross-reference results are bundled with the meeting summary approval:
- Eyal sees a "Cross-Meeting Intelligence" section in the Telegram approval message
- Task status changes show confidence level, assignee, and evidence quote
- Deduplicated tasks show which existing task they matched
- Resolved questions show the answer

When Eyal approves, `_apply_cross_reference_changes()` automatically:
- Updates task statuses in Supabase for approved status changes
- Resolves open questions with `resolved_in_meeting_id` link

### 7. Weekly Digest Integration

The weekly digest now includes a "Cross-Meeting Intelligence This Week" section:
- How many tasks were automatically deduplicated
- Which task statuses were inferred and approved
- How many open questions were resolved

### 8. Time-Weighted RAG Search

Search results now factor in recency. The formula blends RRF score with a recency boost:
```
final_score = 0.7 * rrf_score + 0.3 * recency_boost
recency_boost = 1.0 / (1.0 + days_ago / 30)
```

This means recent meetings rank higher when scores are close, but highly relevant old meetings still surface. The half-life is 30 days — a meeting from 30 days ago gets half the recency boost of today's meeting.

### 9. Parent Chunk Retrieval

When search results are enriched with context (`enrich_chunks_with_context`), the system now also fetches neighboring chunks (chunk_index ± 1). This provides expanded context around each search hit, making answers more complete.

### 10. Query Router

A lightweight pre-classification step at the start of `process_message()` optimizes retrieval:
- **task_status** queries ("status of...", "progress on...", "where are we with...") → pre-fetches relevant tasks + task_mentions
- **decision_history** queries ("when did we decide...", "what was decided about...") → pre-fetches related decisions with meeting context
- **entity_lookup** queries ("tell me about...", "who is...") → future use (v0.4 entity registry)
- **general** queries → no pre-fetching, standard flow

Classification uses simple keyword matching (no LLM call) to avoid latency.

## Database Changes

### New Table: `task_mentions`

```sql
CREATE TABLE task_mentions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    task_id UUID REFERENCES tasks(id) ON DELETE CASCADE,
    meeting_id UUID REFERENCES meetings(id) ON DELETE CASCADE,
    mention_text TEXT NOT NULL,
    implied_status TEXT,              -- 'done', 'in_progress', or NULL
    confidence TEXT DEFAULT 'medium', -- 'high', 'medium', 'low'
    evidence TEXT,                    -- exact quote from transcript
    transcript_timestamp TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

**Migration required:** Run the updated `scripts/setup_supabase.sql` in Supabase SQL Editor (the `CREATE TABLE IF NOT EXISTS` is safe to re-run).

## Design Decisions

1. **Conservative inference:** The system errs on the side of NOT flagging changes. "We're making progress" does not mark a task as done. Only explicit statements or strong implications trigger a change.

2. **Bundled approval:** Cross-reference results are included in the existing meeting summary approval. No separate approval step needed. Eyal sees everything in one message.

3. **Fail-safe dedup:** If the LLM call fails during deduplication, all tasks are treated as new (safe default). No data loss.

4. **Task mentions as audit trail:** Every detected cross-meeting connection creates a `task_mention` record, regardless of whether it leads to a status change. This builds a history of how tasks were discussed across meetings.

5. **Time weighting is additive:** The recency boost is blended with the existing RRF score (70/30 split), not a replacement. This means relevance still dominates, but recency breaks ties.

6. **Query router uses keywords, not LLM:** To avoid adding latency to every user query, classification uses simple string matching. This is fast and handles the common patterns well.

## Phase 2: Tier 2 — Entity Registry, Commitments, Proactive Alerts

Implemented 2026-03-01. Addresses Eyal's three key pain points:
- "Who is Jason Adelman and what have we discussed with him?" (Entity Registry)
- Verbal promises in meetings that nobody tracks (Commitment Tracking)
- Same topics discussed in 3+ meetings without resolution (Proactive Alerts)

### 11. Entity Registry

**Tables:** `entities` (canonical records with aliases) + `entity_mentions` (cross-meeting tracking)

**Extraction approach — Opus piggyback (zero extra cost):**
The main Opus transcript extraction call already sees the full transcript. We enriched its schema to extract stakeholders with type, relationship, and speaker. The entity linker then takes these pre-extracted stakeholders and:
1. Filters out team members and meeting participants
2. Runs a Haiku validation pass (removes generic names like "AWS", "Google")
3. Resolves against existing entities (exact name → alias → partial match)
4. Creates new entity records for unknowns + batch-creates mention records

**Fallback path:** If `pre_extracted` is not available, a standalone Haiku extraction runs on a 16K sample (beginning + middle + end of transcript).

**Agent tools:** `get_entity_info` (lookup + recent mentions) and `get_entity_timeline` (chronological mentions across meetings).

**Seed data:** `scripts/seed_entities.py` pre-populates 7 known entities: Jason Adelman, IIA, Lavazza, Ferrero, Gagauzia, Moldova, Moldova Pilot.

### 12. Commitment Tracking

**Table:** `commitments` with speaker, text, context, implied_deadline, status lifecycle (open → fulfilled/overdue/withdrawn)

**Extraction:** Haiku analyzes the first 8K of each transcript for verbal promises. Clear, actionable commitments only — filters out past-tense statements, general observations, and questions.

**Fulfillment detection:** After extraction, all open commitments are compared against the current transcript. Evidence of fulfillment (explicit statements or strong implications) is flagged with confidence levels.

**Integration:** Both extraction and fulfillment run inside `run_cross_reference()`, stored automatically. Fulfillment goes through Eyal's approval flow.

**Weekly digest:** `get_commitment_scorecard()` shows open commitments grouped by speaker.

### 13. Proactive Alerts

**Four SQL-driven pattern detectors (no LLM calls):**

| Alert | Trigger | Severity |
|-------|---------|----------|
| Overdue task cluster | 3+ overdue tasks for one assignee | High |
| Stale commitment | Open commitment not mentioned in 2+ weeks | Medium |
| Recurring discussion | Entity mentioned in 3+ meetings | Low |
| Open question pileup | 5+ unresolved open questions | Medium |

All thresholds are configurable via `config/settings.py`.

**Scheduler:** `alert_scheduler.py` runs every 12 hours, sends at most one alert batch per day to Eyal via Telegram DM.

**Post-meeting alerts:** `generate_post_meeting_alerts()` runs after each transcript to check if any entity just crossed the 3-meeting threshold.

### 14. Settings Centralization

~25 hardcoded values moved to `config/settings.py` as Pydantic Fields with defaults:
- Search & RAG: `SIMILARITY_THRESHOLD`, `RECENCY_HALFLIFE_DAYS`, `CHUNK_SIZE`, `CHUNK_OVERLAP`
- Alert thresholds: `ALERT_OVERDUE_CLUSTER_MIN`, `ALERT_STALE_COMMITMENT_DAYS`, etc.
- Scheduler intervals: `EMAIL_CHECK_INTERVAL`, `TRANSCRIPT_POLL_INTERVAL`, etc.
- Conversation memory: `CONVERSATION_MAX_MESSAGES`, `CONVERSATION_TTL_MINUTES`

**Key fix:** All schedulers had a module-level constant capture bug (`DEFAULT_X = settings.Y` evaluated at import time, unmockable in tests). Fixed by resolving defaults at runtime inside `__init__`.

### 15. Weekly Entity Health Check

`review_entity_health()` runs as part of the weekly digest:
- Auto-cleans any entities that match team member names
- Flags orphan entities (no mentions)
- Lists entities created this week for Eyal's review
