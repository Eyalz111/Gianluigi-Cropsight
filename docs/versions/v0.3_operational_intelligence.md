# v0.3 — Operational Intelligence

## Overview

v0.3 transforms Gianluigi from a meeting-processing tool into a cross-meeting operational intelligence system. The core idea: when a task from Meeting A gets discussed in Meeting B, Gianluigi should understand the connection, infer status changes, and keep the operational picture up to date — automatically.

**Key context:** CropSight's meetings are founder/management-level. Tasks are high-level strategic commitments (not Jira tickets). Status inference must be nuanced — "we're making progress on the partnership" is not the same as "the partnership is signed."

**Approval strategy:** All inferred changes require Eyal's approval (option A). Additionally, a weekly digest summary of all inferred changes is included in the Sunday digest (option C). This double-layer ensures nothing slips through without human review.

---

## Market Context

Research across Fireflies, Otter, Read.ai, Fellow, Avoma, Granola, Fathom, Lindy.ai, xembly, and others shows that **nobody does automatic cross-meeting task status tracking well**. This is a genuine gap. Most tools offer:
- Per-meeting transcription & summarization (commodity)
- Per-meeting action item extraction (commodity)
- Search across meeting libraries (query-driven, not proactive)

What's missing (and what Gianluigi will do):
- Auto-detecting task completion from natural conversation
- Preventing task duplication across meetings
- Proactive pattern alerts ("discussed 3 times, no resolution")
- Living operational dashboards that update from meeting content

## Architecture: Cross-Reference Stage

The v0.3 architecture adds a **Cross-Reference Stage** to the existing transcript processing pipeline. This is additive — no rewriting of existing code.

```
Current Pipeline (v0.2.1):
  Transcript → Parse → Extract → Store → Embed → Approve → Distribute

v0.3 Pipeline:
  Transcript → Parse → Extract → Store → Embed
                                    |
                           Cross-Reference Stage (NEW)
                           ├── 1. Task Deduplication
                           ├── 2. Task Status Inference
                           ├── 3. Open Question Resolution
                           └── 4. Proactive Alerts
                                    |
                           Approve (now includes ops intelligence) → Distribute
```

### New Supabase Tables

```sql
-- Track where existing tasks are mentioned across meetings
CREATE TABLE task_mentions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    task_id UUID REFERENCES tasks(id) ON DELETE CASCADE,
    meeting_id UUID REFERENCES meetings(id) ON DELETE CASCADE,
    mention_text TEXT NOT NULL,          -- how it was described in this meeting
    implied_status TEXT,                 -- what the conversation implies: pending/in_progress/done
    confidence TEXT DEFAULT 'medium',    -- high/medium/low
    evidence TEXT,                       -- exact quote from transcript
    transcript_timestamp TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_task_mentions_task ON task_mentions(task_id);
CREATE INDEX idx_task_mentions_meeting ON task_mentions(meeting_id);
```

## Tier 1: Core Operational Intelligence (Build First)

### Feature 1: Task Status Inference

**What:** After processing a new transcript, compare the conversation against all open tasks. Use Claude (Sonnet) with chain-of-thought reasoning to infer which tasks had status changes.

**Why:** Eyal's #1 request. Currently tasks live independently — if "set up dev environment" is completed and mentioned in a follow-up meeting, nobody updates the tracker unless they remember.

**How it works:**
1. After transcript extraction, fetch all open tasks from Supabase
2. Send transcript + open tasks to Claude Sonnet with a structured prompt
3. Claude reasons about each task: was it discussed? did the status change?
4. Only flag changes with medium or high confidence
5. Results go through approval flow — Eyal sees:
   ```
   Task Status Updates (inferred from BP Meeting #4):

   [HIGH] "Set up dev environment" (Roye) → DONE
     Evidence: "I've got the dev environment running on AWS" at 23:45

   [MEDIUM] "Draft Moldova timeline" (Paolo) → IN PROGRESS
     Evidence: "I'm working through the timeline" at 31:12

   Approve these changes? [Approve All / Review Each / Skip]
   ```
6. Approved changes update tasks in Supabase + Google Sheets Task Tracker
7. All inferences stored in `task_mentions` table for audit trail

**Complexity considerations:**
- Founder-level tasks are strategic ("secure Series A funding", "close Moldova PoC") — not binary done/not-done
- The LLM must distinguish between "discussing progress" and "confirming completion"
- Confidence thresholds must be conservative: only flag when evidence is clear
- Use `model_agent` (Sonnet) — this requires genuine reasoning, not classification

**Integration point:** New step in `processors/transcript_processor.py` after `store_meeting_data()`, before embedding generation. New file: `processors/cross_reference.py`.

### Feature 2: Task Deduplication

**What:** Before inserting newly extracted tasks, compare against existing tasks to prevent duplicates. When a duplicate is found, create a `task_mention` instead of a new task.

**Why:** Across 4 transcripts in testing, we got 31 tasks. As more meetings are processed, the same strategic tasks will be extracted repeatedly with different wording.

**How it works:**
1. After Claude extracts tasks from a new transcript, collect all existing open tasks
2. Send both lists to Claude Haiku (classification task) asking: for each new task, is it a DUPLICATE of an existing one, an UPDATE to an existing one, or genuinely NEW?
3. For duplicates: create a `task_mention` linking to the existing task
4. For updates: create a `task_mention` + suggest a status change (feeds into Feature 1)
5. For new tasks: insert normally as today
6. Dedup results included in the approval request:
   ```
   Task Deduplication (3 of 8 extracted tasks matched existing):

   "Set up cloud infrastructure" → matches existing "Configure AWS environment" (Roye)
     Action: Link as mention (no new task created)

   "Finalize investor deck" → matches existing "Prepare Series A materials" (Eyal)
     Action: Link as mention, suggest status → IN PROGRESS
   ```

**Integration point:** Modifies the task insertion step in `processors/transcript_processor.py`. Uses `model_simple` (Haiku) for matching since it's a classification task.

---

### Feature 3: Open Question Resolution

**What:** Automatically detect when previously open questions get answered in subsequent meetings.

**Why:** Your schema already has `resolved_in_meeting_id` on `open_questions` but it's never populated. Questions raised in Meeting A often get answered in Meeting B — currently nobody tracks this.

**How it works:**
1. After transcript extraction, fetch all open questions from Supabase
2. Send transcript + open questions to Claude Haiku
3. For each question that appears to be answered, link the resolution:
   ```
   Open Questions Resolved in This Meeting:

   Q: "What's our pricing model for the MVP?" (raised in IP Meeting, Feb 15)
   A: Discussion at 14:20 — "We agreed on per-hectare pricing, $2/ha/season"

   Mark as resolved? [Yes / No]
   ```
4. Approved resolutions update `resolved_in_meeting_id` and `status` in Supabase

**Integration point:** Same cross-reference step as Features 1 and 2. Single LLM call can handle all three.

## Tier 3: RAG & Query Improvements (Build Second)

### Feature 7: Time-Weighted RAG Search

**What:** Boost recent meetings in search results while keeping highly-relevant older ones.

**How:** Modify `_reciprocal_rank_fusion()` in `supabase_client.py`:
```python
def time_weighted_score(rrf_score: float, meeting_date: datetime) -> float:
    days_ago = (datetime.now() - meeting_date).days
    recency_factor = 1.0 / (1.0 + days_ago / 30)  # 30-day half-life
    return 0.7 * rrf_score + 0.3 * recency_factor
```

**Effort:** ~1 hour. Minimal code change.

---

### Feature 8: Parent Chunk Retrieval

**What:** When a search chunk matches, also fetch neighboring chunks (chunk_index +/- 1) for better context.

**How:** In `enrich_chunks_with_context()`, for each matching chunk, also query:
```sql
SELECT chunk_text FROM embeddings
WHERE source_id = :source_id
AND chunk_index IN (:idx - 1, :idx + 1)
ORDER BY chunk_index;
```

Concatenate into expanded context before passing to Claude.

**Effort:** ~2 hours. Simple query addition.

---

### Feature 9: Query Router

**What:** Classify incoming questions and use specialized retrieval strategies.

**How:** Before the agent's tool-use loop, classify the query type with Haiku:
```
Query types:
- "task_status" → prioritize task table + task_mentions
- "entity_lookup" → entity search + timeline
- "decision_history" → decision table + meeting context
- "cross_meeting" → multi-hop search across meetings
- "general" → standard hybrid search (current behavior)
```

Each type pre-loads relevant context before the agent starts reasoning.

**Effort:** ~half a day. Mostly prompt engineering.

### Feature 4: Entity Registry & Cross-Meeting Tracking

**What:** Canonical entity table (people, orgs, projects, technologies) with aliases. During transcript processing, match mentions to existing entities using Haiku. Enables "What's our full history with Gagauzia?" queries.

**Schema:**
```sql
CREATE TABLE entities (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    canonical_name TEXT NOT NULL,
    entity_type TEXT NOT NULL,       -- person, organization, project, technology
    aliases TEXT[] DEFAULT '{}',
    metadata JSONB DEFAULT '{}',
    first_seen_meeting_id UUID REFERENCES meetings(id),
    created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE entity_mentions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    entity_id UUID REFERENCES entities(id),
    meeting_id UUID REFERENCES meetings(id),
    mention_text TEXT NOT NULL,
    context TEXT,
    speaker TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```

**Value:** Enriches meeting prep ("Last time you met Jason, key items were X, Y, Z"). Enables stakeholder history tracking. Makes weekly digests richer.

**Why deferred:** Task status inference and dedup deliver more immediate value. Entity registry can build on top of the `task_mentions` pattern once it's proven.

### Feature 5: Proactive Cross-Meeting Alerts

**What:** After processing a transcript, surface patterns via Telegram:
- "This is the 3rd meeting where Series A timeline was discussed with no resolution"
- "Roye has 5 overdue tasks from the last 2 weeks"
- "Jason committed to the LOI on Jan 15, still pending as of Feb 28"

**Why deferred:** Requires Feature 4 (entity registry) to reliably detect recurring topics across meetings. Can be built incrementally — start with simple task-based alerts (overdue counts), add entity-based alerts later.

---

### Feature 6: Commitment Tracking Dashboard

**What:** Dedicated tracking of who committed to what, when, and whether it was fulfilled. Separate from tasks — commitments are verbal promises made in meetings that may or may not map to formal tasks.

**Why deferred:** Task status inference (Feature 1) covers most of the value. Commitments add a second tracking dimension that may be confusing for a 4-person team.

5. **Tests throughout** — each feature gets tested as it's built
6. **Weekly digest integration last** — ties everything together

## Design Principles

1. **Additive, not rewriting:** Every feature plugs into the existing pipeline. No restructuring.
2. **Approval as safety net:** All inferred changes go through Eyal before being applied.
3. **Conservative inference:** Only flag status changes with clear evidence. "Discussed" is not "done."
4. **LLM tier optimization:** Haiku for classification ($0.001/call), Sonnet for reasoning ($0.01/call).
5. **PostgreSQL-native:** No new infrastructure. Everything in Supabase.
6. **Audit trail:** Every inference stored in `task_mentions` with evidence and confidence.
7. **Founder-level sensitivity:** These are strategic tasks, not Jira tickets. The LLM must understand the difference between "making progress" and "completed."
