# Gianluigi v0.5 — "Gianluigi Goes to Work"

**Goal:** Ship to Cloud Run, cut LLM costs 35-50%, improve output quality, add document ingestion, and harden for daily team use.

**Status:** v0.4.1 complete (499 tests, all live-tested). No deployment yet — still running locally.

---

## Current State Audit

### What's Working (v0.1–v0.4.1)
- Full transcript pipeline: Tactiq → Opus extraction → Supabase + embeddings
- Approval flow with conversational editing (Telegram + email)
- Distribution: .md + .docx to Drive, tasks to Sheets, follow-ups, stakeholder updates
- Cross-meeting intelligence: task dedup, status inference, open Q resolution
- Entity registry + commitment tracking + proactive alerts
- Calendar classification memory (persistent, fuzzy matching)
- Reprocess command, commitment dashboard in Sheets
- Hybrid RAG: semantic + full-text + time-weighted + parent chunks
- Query router, conversation memory, inbound guardrails
- 499 tests passing

### What's Not Deployed
- Cloud Run infra is built (Dockerfile, cloudbuild.yaml, health server) but never deployed
- `ENVIRONMENT=development` gates all distribution — production switch needed

### What's Missing for Daily Use
1. No document ingestion (slides, PDFs, strategy docs in Drive)
2. Writing quality is functional but template-heavy — not engaging to read
3. No cost monitoring or prompt caching on most LLM calls
4. Telegram UX is basic — no search within bot, no meeting browse
5. No scheduled error recovery (orphan cleanup, retry failed processing)
6. No onboarding path — new team member can't self-serve context

### Track B: Cost Optimization (Target: 35-50% reduction)

Current estimated monthly cost: $3.45–$4.91. Target: $2.00–$3.00.

#### B1 — Add prompt caching to all system prompts
**Current state:** Only `agent.py` (line 66) uses `cache_control: {"type": "ephemeral"}`.

**Missing caching on:**
| Call site | File:Line | Model | System prompt size |
|-----------|-----------|-------|-------------------|
| Transcript extraction | `transcript_processor.py:458` | Opus | ~6,000 tokens |
| Task deduplication | `cross_reference.py:112` | Haiku | ~500 tokens |
| Status inference | `cross_reference.py:245` | Sonnet | ~800 tokens |
| Commitment extraction | `cross_reference.py:328` | Haiku | ~400 tokens |
| Open Q resolution | `cross_reference.py:538` | Haiku | ~400 tokens |
| Commitment fulfillment | `cross_reference.py:613` | Haiku | ~400 tokens |
| Entity extraction | `entity_extraction.py:288` | Haiku | ~600 tokens |
| Entity validation | `entity_extraction.py:372` | Haiku | ~400 tokens |
| Meeting prep | `meeting_prep.py` | Sonnet | ~1,200 tokens |
| Weekly digest | `weekly_digest.py` | Sonnet | ~1,000 tokens |
| Edit processing | `approval_flow.py:690` | Haiku | ~300 tokens |
| Distribution formatting | `approval_flow.py:754` | Sonnet | ~500 tokens |

**Savings:** Cached prompts cost 90% less on subsequent calls. For the transcript pipeline (which fires all cross-ref + entity calls sequentially), a single Anthropic client instance with cached system blocks could save ~4,000 cached input tokens per meeting.

**Implementation:**
- Create a helper `_cached_message(system_prompt, user_message, model)` in a new `core/llm.py` module
- All LLM calls route through this helper
- Centralizes model selection, caching, error handling, and future token tracking

#### B2 — Fix model tier for status inference
`cross_reference.py:245` uses `model_agent` (Sonnet) for `infer_task_status_changes`. This is a classification task — can use `model_simple` (Haiku) like the other cross-ref calls.

**Savings:** ~5-8% of total monthly cost (status inference runs every meeting with full transcript).

#### B3 — Token usage dashboard
Add a `token_usage` table to Supabase:
```sql
CREATE TABLE token_usage (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    call_site TEXT NOT NULL,      -- e.g., 'transcript_extraction', 'task_dedup'
    model TEXT NOT NULL,
    input_tokens INTEGER,
    output_tokens INTEGER,
    cached_tokens INTEGER DEFAULT 0,
    cost_usd NUMERIC(10, 6),
    meeting_id UUID REFERENCES meetings(id),
    created_at TIMESTAMPTZ DEFAULT NOW()
);
```
- Log from the centralized `_cached_message` helper
- Weekly digest includes a "Cost Summary" line: "This week: X API calls, $Y.YY total"
- `/cost` Telegram command for Eyal: shows last 7 days breakdown by model tier

**Estimated effort:** 1 session

### Track C: Writing Quality Enhancement

Current output is structured and accurate but reads like a template fill-in. Goal: make summaries that the team actually *wants* to read.

#### C1 — Discussion summary upgrade
**Problem:** The `discussion_summary` in extraction prompt just says "2-4 paragraphs." Results are often flat recaps.

**Enhancement — add structured narrative guidance:**
```
6. Write a DISCUSSION SUMMARY
   - Opening paragraph: What was the meeting's purpose and key outcome?
   - Body paragraphs: Group by theme, not chronology. Connect related
     discussions even if they happened at different points in the meeting.
   - Closing paragraph: What's the overall trajectory? Are things on track,
     at risk, or pivoting?
   - Write for a reader who missed the meeting — they should understand
     both WHAT happened and WHY it matters.
   - Use active voice. Be specific, not vague.
   - BAD: "The team discussed the Moldova pilot and various challenges."
   - GOOD: "The Moldova pilot timeline was the central tension point:
     delivery is on track technically, but client expectations around
     accuracy documentation need to be managed before the next milestone."
```

#### C2 — Executive summary line
Add a one-line "TLDR" field to the extraction schema:
```json
"executive_summary": "One sentence capturing the meeting's key outcome"
```
- Displayed at the top of Telegram approval messages
- First line in email summaries
- Helps Eyal decide approval priority at a glance

#### C3 — Smarter task descriptions
Current task extraction produces bare descriptions like "Write accuracy abstract." Enhancement:
```
TASK EXTRACTION ENHANCEMENT:
- Each task should answer: WHO does WHAT by WHEN and WHY.
- BAD: "Write accuracy abstract"
- GOOD: "Write 1-page accuracy abstract documenting model performance
  benchmarks — needed before the Feb 27 meeting so the team can discuss
  client-facing accuracy claims"
- Include the business context, not just the action.
```

#### C4 — Meeting prep quality
Current prep docs list context but don't synthesize. Add guidance:
```
SYNTHESIS INSTRUCTIONS:
- Don't just list past decisions — explain which ones are still
  relevant and which may need revisiting.
- For each attendee, note their last stated position on key topics.
- Suggest 2-3 "conversation starters" — questions that could
  move stuck items forward.
- If there are overdue tasks for any attendee, flag diplomatically.
```

#### C5 — Email formatting
Currently emails contain a link to the Drive document. Enhancement:
- Include the executive summary line in the email body
- Include the task table directly in the email (HTML table)
- Add "Your action items" section personalized per recipient (filter tasks by assignee)
- This makes the email useful without clicking through to Drive

**Estimated effort:** 1 session (mostly prompt engineering + template changes)

### Track D: Document Ingestion Pipeline

**Problem:** Gianluigi only processes transcripts. CropSight has strategy docs, pitch decks, research papers, and MOUs in Drive that should be part of institutional memory.

#### D1 — Document watcher enhancement
The `document_watcher` scheduler already exists but only handles simple text files. Enhance:

- **New Drive folder:** `Documents/` (already in the Drive structure plan)
- **Supported formats:** PDF, DOCX, PPTX, plain text
- **Processing:**
  1. Detect new files in Documents folder
  2. Extract text (python-docx for .docx, python-pptx for .pptx, PyPDF2/pdfplumber for .pdf)
  3. Summarize with Claude Haiku (cheap — documents are usually shorter than transcripts)
  4. Chunk and embed for RAG search
  5. Store in `documents` table (already exists in schema)
  6. Notify Eyal via Telegram: "Ingested new document: {title}"

#### D2 — Document type classification
Tag documents by type for better retrieval:
- `strategy` — strategy docs, competitive analyses
- `legal` — MOUs, NDAs, contracts
- `technical` — research papers, technical specs
- `pitch` — pitch decks, investor materials
- `client` — client-facing materials, proposals

The query router can use document type to improve search relevance.

#### D3 — Slide deck handling (PPTX)
Slides need special treatment:
- Extract text from each slide
- Preserve slide order as chunk order
- Speaker notes → separate metadata field
- Diagrams/charts → note "[Diagram: title]" placeholder (no image processing in v0.5)

#### D4 — Cross-reference documents with meetings
When a document is mentioned in a meeting ("the pitch deck we sent to Lavazza"), link it:
- Entity extraction already catches organization names
- Add document matching: when an entity mention context includes document-related keywords ("deck", "doc", "paper", "proposal"), search `documents` table for matches
- Display linked documents in meeting prep

**Dependencies:** `python-pptx`, `pdfplumber` (add to requirements.txt)

**Estimated effort:** 2 sessions

### Track F: Production Hardening

#### F1 — Retry logic for transient failures
- Google API calls (Drive, Sheets, Calendar, Gmail) should retry 3x with exponential backoff
- Supabase calls: retry on connection errors, not on constraint violations
- LLM calls: retry on 429 (rate limit) and 529 (overloaded), not on 400 (bad request)
- Use a simple decorator: `@retry(max_attempts=3, backoff=2)`

#### F2 — Orphan cleanup scheduler
Run daily:
- Meetings with `approval_status='pending'` older than 7 days → notify Eyal
- Embeddings with no matching meeting → delete
- Tasks with no meeting and `status='pending'` older than 30 days → flag
- Pending approvals past `auto_publish_at` that weren't published → retry

#### F3 — Structured logging
Replace `logger.info(f"...")` patterns with structured JSON logging:
- Add `meeting_id`, `file_id`, `action` fields to log entries
- When running on Cloud Run, these become queryable in Cloud Logging
- Keep human-readable format for local dev

#### F4 — Error alerting
When critical errors occur (pipeline failure, API auth expiry, DB connection loss):
- Send Telegram message to Eyal: "Gianluigi error: {description}"
- Log to `audit_log` with `action='critical_error'`
- Don't spam — deduplicate same error within 1 hour

**Estimated effort:** 1 session
